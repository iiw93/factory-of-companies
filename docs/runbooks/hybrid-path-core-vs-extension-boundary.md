# Hybrid Path Core vs Extension Boundary

## 1. Purpose
Define the explicit boundary between the Hybrid Path target core architecture and the Factory-of-Companies extension layer in planning/governance terms only.
This artifact is non-authorizing and does not activate runtime or technical implementation.

## 2. Core Layer
- Company-scoped control-plane principles define the target core architecture boundary.
- Governance/reopen discipline remains mandatory: exact scope naming, explicit approvals, and one-track-only control.
- Execution boundary rules: architecture/planning outputs do not imply runtime activation or implementation authority.
- Scenario-01 runtime authority guardrails remain in force as the only authoritative implemented runtime path unless separately changed by explicit decision.

## 3. Extension Layer
- Multi-company coordination concerns that operate across company boundaries.
- Shared-infrastructure orchestration for cross-company platform operations.
- Cross-company control surfaces beyond single-company core control-plane scope.
- Factory-of-Companies-specific extensions that are outside the target core architecture boundary.

## 4. Boundary Rules
- No leakage of extension concerns into core by implication; any extension concern must be explicitly classified as extension.
- No runtime activation from architecture/planning work; documentation outputs are governance/planning artifacts only.
- No blocked-track activation without a separate explicit reopen decision naming exact scope.
- Preserve one-track-only discipline; scope may not silently broaden to adjacent tracks or surfaces.

## 5. Decision Table
| Concern | Core / Extension | Why | Current status |
| --- | --- | --- | --- |
| Company-scoped control-plane principles | Core | Defines target architecture baseline under Hybrid Path. | Active governance direction. |
| Governance/reopen discipline | Core | Controls authorization boundaries and scope safety. | Active and required. |
| Scenario-01 runtime authority guardrails | Core | Preserves authoritative implemented runtime path. | Active and unchanged. |
| Multi-company coordination | Extension | Cross-company orchestration is outside single-company core boundary. | Not reopened for implementation. |
| Shared-infrastructure orchestration | Extension | Shared platform operations are Factory-specific extension concerns. | Not reopened for implementation. |
| Cross-company control surfaces | Extension | Exceeds core company-scoped control-plane boundary. | Not reopened for implementation. |

## 6. Risks if Boundary Blurs
- Implicit scope expansion from planning artifacts into unauthorized implementation.
- Authority weakening that could erode Scenario-01 runtime guardrails.
- Premature blocked-track leakage without explicit reopen discipline.
- Governance drift that conflates core architecture direction with extension execution work.

## 7. Constraint on Future Technical Reopen
Any future technical reopen must explicitly preserve this core-vs-extension boundary and name exact technical scope that respects it.
This file does not authorize any technical reopen, runtime implementation, blocked-track activation, contract change, or acceptance expansion.
